Домашнее задание
1.Придумать клиент серверную многопользовательскую систему
2.Выбрать для нее архитектуру
3.Кратко описать:
-Саму систему ее архитектуру чем обусловлен выбор архитектуры
-Какие компоненты есть в вашей архитектуре за что каждый из них отвечает
-Как компоненты взаимодействуют между собой
-Как ваша архитектура решает проблемы производительности надежности и безопасности

1. Учет трудоемкости выполнения работ.
Задача сбора с работников данных о длительности выполнения работ (человеко-часы).
Хранить данные о трудоёмкости и выдавать данные руководителю по запросу. При этом данные 
по трудоемкостям хранятся раздельно для разных крупных объектов (например, 
проектирование заводов в разных городах).
2. Двухуровневая архитектура явно не подходит, так как потребуются сервера 
по проектам и базы данных для хранения информации, так же серверы приложений 
и уровень пользовательского звена. Поэтому трехуровневая. 
Так же из соображений безопасности не будет прямого доступа пользователям к БД.
3.Трехуровневая архитектура «клиент-сервер » 

Краткое описание.
1. Обмен данными с пользователем. На этом уровне ввод данных по форме, 
а также запрос данных пользователем о ранее введенных данных в установленных 
формах (например, средняя трудоемкость по данному виду работ по выбранному 
проекту или по всем проектам в целом)
2. Уровень обработки (логика). На данном уровне обработка данных пользователя 
и обработка запросов клиента(назовем его Сервер 1), взаимодействие Сервера 1 с Серверами Проектов. 
3. Серверы Проектов записываю и читают в БД конкретного проекта. Данное решение 
позволит нам хранить БД на различных физических носителях.

Безопасность. 
Авторизация пользователя, тогда ввод и учет данных ведется исключительно 
по данному пользователю, это даст относительную достоверность данных.
На Сервере 1 проверяем входные данные (корректируем или отклоняем перед записью)
это обеспечит целостность и корректность данных. Логика не доступна пользователю, 
поэтому повлиять на ход обработки он не может.
Сервер Проекта взаимодействует с Сервером 1 и БД без вмешательства пользователя.
Распределение данных по различным БД повышает устойчивость системы.

Производительность.
Разделение проектов по различным БД само по себе сокращает объем 
обрабатываем информации.
Так же на основе анализа запросов пользователей разумно на уровне 
Серверов Проектов создать Сервер Частых запросов и БД в которую, 
например по расписанию, агрегатируются часто запрашиваемые данные (для задачи
получения осреднённой трудоемкости по типу работы при большом количестве работ
погрешность вычисления будет мала и данные часто будут удовлетворять потребности
пользователя).
